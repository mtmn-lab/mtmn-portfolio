**📊 単純平均の罠と「加重平均」の重要性**

# **1. なぜ「％」をそのまま足してはいけないのか？（加重平均）**

データ分析において、最も多いミスが**「比率（％）の単純平均」**です。

**❌ 間違った集計（単純平均）**

- **A店舗：** 売上 1,000万円、利益率 10%
- **B店舗：** 売上 10万円、利益率 90%
- **単純平均：** (10\% + 90\%) / 2 = \mathbf{50\%}
- **結果：** 全社の利益率は50%だ！ → **大間違い。**

**⭕️ 正しい集計（加重平均）**

一度「実数（利益額）」に戻して合計し、全体の売上で割ります。

- **A店舗の利益：** 1,000 \times 0.1 = 100万円
- **B店舗の利益：** 10 \times 0.9 = 9万円
- **全社の利益率：** (100 + 9) / (1,000 + 10) = \mathbf{約10.8\%}

**結論：** 規模（母数）が異なるデータをまとめる時は、必ず「掛けて実数に戻してから、最後にもう一度割る」ステップが必要です。

# **2. 実務で要注意な「計算の罠」リスト**

実務でのアンチョコとして、以下のケースは特に注意してください。

**① 「0除算（Division by Zero）」のエラー**

データが空だったり、分母が0になるとレポートが壊れます。

- **SQL:** NULLIF(分母, 0) を使って回避する。
- **DAX:** DIVIDE(分子, 分母, 0) を使う（第3引数に0を指定）。

**② 「NULL」の扱い**

SQLの AVG や SUM は通常 NULL を無視しますが、計算式の中で + NULL をすると結果がすべて NULL になります。

- **対策:** COALESCE(カラム名, 0) で NULL を 0 に置換してから計算する癖をつける。

**③ 「平均の平均」問題**

日次の平均値をさらに月次で平均すると、日ごとのデータ件数が違う場合に数字が狂います。

- **鉄則:** 常に「生データ（明細）」のレベルで合計を出してから、最後に割る。

**④ 重複データのカウント（COUNT vs DISTINCT COUNT）**

キャンペーン分析などで、一人の顧客が何度も購入している場合、単純な COUNT だと「人数」ではなく「件数」になってしまいます。

- **SQL:** COUNT(DISTINCT user_id)
- **DAX:** DISTINCTCOUNT(user_id)

# **3. SQL / DAX 実装サンプル**

NSS（学生調査）のように「比率」と「回答数」が分かれている場合の標準的な書き方です。

/* SQL: 全体の加重平均を出す */
SELECT
category,
SUM(score * sample) / SUM(sample) AS weighted_avg
FROM
survey_data
GROUP BY
category;

/* DAX: メジャーで利益率を出す (これも加重平均) */
Total Profit Margin =
DIVIDE(
SUM('Sales'[Profit]),
SUM('Sales'[Sales Amount]),
0
)
